MXBean specification

  83 
  84     The MXBean concept provides a simple way to code an MBean
  85       that only references a predefined set of types, the ones defined
  86       by {@link javax.management.openmbean}.  In this way, you can be
  87       sure that your MBean will be usable by any client, including
  88       remote clients, without any requirement that the client have
  89       access to model-specific classes representing the types
  90       of your MBeans.
  91 
  92     The concepts are easier to understand by comparison with the
  93       Standard MBean concept.  Here is how a managed object might be
  94       represented as a Standard MBean, and as an MXBean:
  95 
  96     

  97       
  98         Standard MBeanMXBean
  99       
 100       
 101         

 102 public interface MemoryPoolMBean {
 103     String getName();
 104     MemoryUsage getUsage();
 105     // ...
 106 }
 107           
 108         

 109 public interface MemoryPoolMXBean {
 110     String getName();
 111     MemoryUsage getUsage();
 112     // ...
 113 }
 114           
 115       
 116     
 117 
 118     As you can see, the definitions are very similar.  The only
 119       difference is that the convention for naming the interface is to use
 120       SomethingMXBean for MXBeans, rather than
 121       SomethingMBean for Standard MBeans.
 122 
 123     In this managed object, there is an attribute called
 124       Usage of type {@link MemoryUsage}.  The point of an
 125       attribute like this is that it gives a coherent snapshot of a set
 126       of data items.  For example, it might include the current amount
 127       of used memory in the memory pool, and the current maximum of the
 128       memory pool.  If these were separate items, obtained with separate
 129       {@link MBeanServer#getAttribute getAttribute} calls, then we could
 130       get values seen at different times that were not consistent.  We
 131       might get a used value that was greater than the
 132       max value.
 133 
 134     So, we might define MemoryUsage like this:
 135 
 136     

 137       
 138         Standard MBeanMXBean
 139       
 140       
 141         

 142 public class MemoryUsage implements Serializable {
 143     // standard JavaBean conventions with getters
 144 
 145     public MemoryUsage(long init, long used,
 146                        long committed, long max) {...}
 147     long getInit() {...}
 148     long getUsed() {...}
 149     long getCommitted() {...}
 150     long getMax() {...}
 151 }
 152           
 153         

 154 public class MemoryUsage {
 155     // standard JavaBean conventions with getters
 156     @ConstructorProperties({"init", "used", "committed", "max"})
 157     public MemoryUsage(long init, long used,
 158                        long committed, long max) {...}
 159     long getInit() {...}
 160     long getUsed() {...}
 161     long getCommitted() {...}
 162     long getMax() {...}
 163 }
 164           
 165       
 166     
 167 
 168     The definitions are the same in the two cases, except
 169       that with the MXBean, MemoryUsage no longer needs to
 170       be marked Serializable (though it can be).  On
 171       the other hand, we have added a {@code @ConstructorProperties} annotation
 172       to link the constructor parameters to the corresponding getters.
 173       We will see more about this below.
 174 
 175     MemoryUsage is a model-specific class.
 176       With Standard MBeans, a client of the MBean Server cannot access the
 177       Usage attribute if it does not know the class
 178       MemoryUsage.  Suppose the client is a generic console
 179       based on JMX technology.  Then the console would have to be
 180       configured with the model-specific classes of every application it
 181       might connect to.  The problem is even worse for clients that are
 182       not written in the Java language.  Then there may not be any way
 183       to tell the client what a MemoryUsage looks like.
 184 
 185     This is where MXBeans differ from Standard MBeans.  Although we
 186       define the management interface in almost exactly the same way,
 187       the MXBean framework converts model-specific classes into
 188       standard classes from the Java platform.  Using arrays and the
 189       {@link javax.management.openmbean.CompositeData CompositeData} and
 190       {@link javax.management.openmbean.TabularData TabularData} classes
 191       from the standard {@link javax.management.openmbean} package, it
 192       is possible to build data structures of arbitrary complexity
 193       using only standard classes.
 194 
 195     This becomes clearer if we compare what the clients of the two
 196       models might look like:
 197 
 198     

 199       
 200         Standard MBeanMXBean
 201       
 202       
 203         

 204 String name = (String)
 205     mbeanServer.{@link MBeanServer#getAttribute
 206     getAttribute}(objectName, "Name");
 207 MemoryUsage usage = (MemoryUsage)
 208     mbeanServer.getAttribute(objectName, "Usage");
 209 long used = usage.getUsed();
 210           
 211         

 212 String name = (String)
 213     mbeanServer.{@link MBeanServer#getAttribute
 214     getAttribute}(objectName, "Name");
 215 {@link CompositeData} usage = (CompositeData)
 216     mbeanServer.getAttribute(objectName, "Usage");
 217 long used = (Long) usage.{@link CompositeData#get get}("used");
 218           
 219     
 220 
 221     For attributes with simple types like String, the
 222       code is the same.  But for attributes with complex types, the
 223       Standard MBean code requires the client to know the model-specific
 224       class MemoryUsage, while the MXBean code requires no
 225       non-standard classes.
 226 
 227     The client code shown here is slightly more complicated for the
 228       MXBean client.  But, if the client does in fact know the model,
 229       here the interface MemoryPoolMXBean and the
 230       class MemoryUsage, then it can construct a
 231       proxy.  This is the recommended way to interact with
 232       managed objects when you know the model beforehand, regardless
 233       of whether you are using Standard MBeans or MXBeans:
 234 
 235     

 236       
 237         Standard MBeanMXBean
 238       
 239       
 240         

 241 MemoryPoolMBean proxy =
 242     JMX.{@link JMX#newMBeanProxy(MBeanServerConnection, ObjectName,
 243               Class) newMBeanProxy}(
 244         mbeanServer,
 245         objectName,
 246         MemoryPoolMBean.class);
 247 String name = proxy.getName();
 248 MemoryUsage usage = proxy.getUsage();
 249 long used = usage.getUsed();
 250           
 251         

 252 MemoryPoolMXBean proxy =
 253     JMX.{@link JMX#newMXBeanProxy(MBeanServerConnection, ObjectName,
 254               Class) newMXBeanProxy}(
 255         mbeanServer,
 256         objectName,
 257         MemoryPoolMXBean.class);
 258 String name = proxy.getName();
 259 MemoryUsage usage = proxy.getUsage();
 260 long used = usage.getUsed();

Definition of an MXBean

 319 
 320     An MXBean is a kind of MBean.  An MXBean object can be
 321       registered directly in the MBean Server, or it can be used as an
 322       argument to {@link StandardMBean} and the resultant MBean
 323       registered in the MBean Server.
 324 
 325     When an object is registered in the MBean Server using the
 326       {@code registerMBean} or {@code createMBean} methods of the
 327       {@link MBeanServer} interface, the object's class is examined
 328       to determine what type of MBean it is:
 329 
 330     
    
 331       
  • If the class implements the interface {@link DynamicMBean}
    • 
 332         then the MBean is a Dynamic MBean.  Note that the class
 333         {@code StandardMBean} implements this interface, so this
 334         case applies to a Standard MBean or MXBean created using
 335         the class {@code StandardMBean}.
 336 
 337       
  • Otherwise, if the class matches the Standard MBean naming
    • 
 338         conventions, then the MBean is a Standard MBean.
 339 
 340       
  • Otherwise, it may be an MXBean.  The set of interfaces
    • 
 341         implemented by the object is examined for interfaces that:
 342 
 343         
      
 344           
    • have a class name SMXBean where
      • 
 345             S is any non-empty string, and
 346             do not have an annotation {@code @MXBean(false)}; and/or
 347           
    • have an annotation {@code @MXBean(true)}
      • 
 348             or just {@code @MXBean}.
 349         
 350 
 351         If there is exactly one such interface, or if there is one
 352         such interface that is a subinterface of all the others, then
 353         the object is an MXBean.  The interface in question is the
 354         MXBean interface.  In the example above, the MXBean
 355         interface is {@code MemoryPoolMXBean}.
 356 
 357       
    • If none of these conditions is met, the MBean is invalid and
      • 
 358         the attempt to register it will generate {@link
 359         NotCompliantMBeanException}.
 360     
 361 
 362     Every Java type that appears as the parameter or return type of a
 363       method in an MXBean interface must be convertible using
 364       the rules below.  Additionally, parameters must be
 365       reconstructible as defined below.
 366 
 367     An attempt to construct an MXBean that does not conform to the
 368       above rules will produce an exception.
 369

 371     
      Naming conventions
      
 372 
 373     The same naming conventions are applied to the methods in an
 374       MXBean as in a Standard MBean:
 375 
 376     
        
 377       
      1. A method T getN(), where
        2. 
 378         T is a Java type (not void)
 379         and N is a non-empty string, specifies
 380         that there is a readable attribute called
 381         N.  The Java type and Open type of the
 382         attribute are determined by the mapping rules below.
 383         The method {@code final Class getClass()} inherited from {@code
 384         Object} is ignored when looking for getters.
 385 
 386       
      2. A method boolean isN() specifies that
        3. 
 387         there is a readable attribute called N
 388         with Java type boolean and Open type
 389         SimpleType.Boolean.
 390 
 391       
      3. A method void setN(T x)
        4. 
 392         specifies that there is a writeable attribute called
 393         N.  The Java type and Open type of the
 394         attribute are determined by the mapping rules below.  (Of
 395         course, the name x of the parameter is
 396         irrelevant.)
 397 
 398       
      4. Every other method specifies that there is an operation with
        5. 
 399         the same name as the method.  The Java type and Open type of the
 400         return value and of each parameter are determined by the mapping
 401         rules below.
 402     
 403 
 404     The rules for getN and
 405       isN collectively define the notion of a
 406       getter.  The rule for setN defines
 407       the notion of a setter.
 408 
 409     It is an error for there to be two getters with the same name, or
 410       two setters with the same name.  If there is a getter and a setter
 411       for the same name, then the type T in both
 412       must be the same.  In this case the attribute is read/write.  If
 413       there is only a getter or only a setter, the attribute is
 414       read-only or write-only respectively.

        Type mapping rules
        
 418 
 419     An MXBean is a kind of Open MBean, as defined by the {@link
 420       javax.management.openmbean} package.  This means that the types of
 421       attributes, operation parameters, and operation return values must
 422       all be describable using Open Types, that is the four
 423       standard subclasses of {@link javax.management.openmbean.OpenType}.
 424       MXBeans achieve this by mapping Java types into Open Types.
 425 
 426     For every Java type J, the MXBean mapping is described
 427       by the following information:
 428 
 429     
          
 430       
        • The corresponding Open Type, opentype(J).  This is
          • 
 431         an instance of a subclass of {@link
 432         javax.management.openmbean.OpenType}.
 433       
        • The mapped Java type, opendata(J), which is
          • 
 434         always the same for any given opentype(J).  This is a Java
 435         class.
 436       
        • How a value is converted from type J to type
          • 
 437         opendata(J).
 438       
        • How a value is converted from type opendata(J) to
          • 
 439         type J, if it can be.
 440     
 441 
 442     For example, for the Java type {@code List}:
 443 
 444     
            
 445       
          • The Open Type, opentype({@code
            • 
 446         List}), is {@link ArrayType}(1, {@link
 447           SimpleType#STRING}), representing a 1-dimensional
 448           array of Strings.
 449       
          • The mapped Java type, opendata({@code
            • 
 450         List}), is {@code String[]}.
 451       
          • A {@code List} can be converted to a {@code String[]}
            • 
 452           using {@link List#toArray(Object[]) List.toArray(new
 453           String[0])}.
 454       
          • A {@code String[]} can be converted to a {@code List}
            • 
 455           using {@link Arrays#asList Arrays.asList}.
 456     
 457 
 458     If no mapping rules exist to derive opentype(J) from
 459       J, then J cannot be the type of a method
 460       parameter or return value in an MXBean interface.
 461 
 462     
            If there is a way to convert
            
 463       opendata(J) back to J then we say that J is
 464       reconstructible.  All method parameters in an MXBean
 465       interface must be reconstructible, because when the MXBean
 466       framework is invoking a method it will need to convert those
 467       parameters from opendata(J) to J.  In a proxy
 468       generated by {@link JMX#newMXBeanProxy(MBeanServerConnection,
 469       ObjectName, Class) JMX.newMXBeanProxy}, it is the return values
 470       of the methods in the MXBean interface that must be
 471       reconstructible.
 472 
 473     Null values are allowed for all Java types and Open Types,
 474       except primitive Java types where they are not possible.  When
 475       converting from type J to type opendata(J) or
 476       from type opendata(J) to type J, a null value is
 477       mapped to a null value.
 478 
 479     The following table summarizes the type mapping rules.
 480 
 481

 485         opendata(J)

 490             (the 8 primitive Java types)
 491           {@code SimpleType.INTEGER},
            
 492             {@code SimpleType.BOOLEAN}, etc
 493           {@code Integer}, {@code Boolean}, etc
            
 494             (the corresponding boxed types)
 495         
 496         
 497           {@code Integer}, {@code ObjectName}, etc
            
 498             (the types covered by {@link SimpleType})
 499           the corresponding {@code SimpleType}
 500           J, the same type
 501         
 502         
 503           {@code int[]} etc
            
 504             (a one-dimensional array with
            
 505             primitive element type)
 506           {@code ArrayType.getPrimitiveArrayType(int[].class)} etc
 507           J, the same type
 508         
 509           E{@code []}
            
 510             (an array with non-primitive element type E;
 511               this includes {@code int[][]}, where E is {@code int[]})
 512           {@code ArrayType.getArrayType(}opentype(E){@code )}
 513           opendata(E){@code []}
 514

 536         
 537

 556     
            Mappings for primitive types
            
 557 
 558     The 8 primitive Java types
 559       ({@code boolean}, {@code byte}, {@code short}, {@code int}, {@code
 560       long}, {@code float}, {@code double}, {@code char}) are mapped to the
 561       corresponding boxed types from {@code java.lang}, namely {@code
 562       Boolean}, {@code Byte}, etc.  The Open Type is the corresponding
 563       {@code SimpleType}.  Thus, opentype({@code
 564       long}) is {@code SimpleType.LONG}, and
 565       opendata({@code long}) is {@code
 566       java.lang.Long}.
 567 
 568     An array of primitive type such as {@code long[]} can be represented
 569       directly as an Open Type.  Thus, openType({@code
 570       long[]}) is {@code
 571       ArrayType.getPrimitiveArrayType(long[].class)}, and
 572       opendata({@code long[]}) is {@code
 573       long[]}.
 574 
 575     In practice, the difference between a plain {@code int} and {@code
 576       Integer}, etc, does not show up because operations in the JMX API
 577       are always on Java objects, not primitives.  However, the
 578       difference does show up with arrays.
 579 
 580 
 581     
            Mappings for collections ({@code List<}E{@code >} etc)
            
 582 
 583     A {@code List<}E{@code >} or {@code
 584       Set<}E{@code >}, such as {@code List} or {@code
 585         Set}, is mapped in the same way as an array of the
 586           same element type, such as {@code String[]} or {@code
 587           ObjectName[]}.
 588 
 589     A {@code SortedSet<}E{@code >} is also mapped in the
 590       same way as an E{@code []}, but it is only convertible if
 591       E is a class or interface that implements {@link
 592       java.lang.Comparable}.  Thus, a {@code SortedSet} or
 593         {@code SortedSet} is convertible, but a {@code
 594           SortedSet} or {@code SortedSet>} is not.  The
 595                 conversion of a {@code SortedSet} instance will fail with an
 596                 {@code IllegalArgumentException} if it has a
 597                 non-null {@link java.util.SortedSet#comparator()
 598                 comparator()}.
 599 
 600     A {@code List<}E{@code >} is reconstructed as a
 601       {@code java.util.ArrayList<}E{@code >};
 602       a {@code Set<}E{@code >} as a
 603       {@code java.util.HashSet<}E{@code >};
 604       a {@code SortedSet<}E{@code >} as a
 605       {@code java.util.TreeSet<}E{@code >}.
 606 
 607 
 608     
            Mappings for maps ({@code Map<}K,V{@code >} etc)
            
 609 
 610     A {@code Map<}K,V{@code >} or {@code
 611       SortedMap<}K,V{@code >}, for example {@code
 612       Map}, has Open Type {@link TabularType} and is mapped
 613         to a {@link TabularData}.
 614         The {@code TabularType} has two items called {@code key} and
 615         {@code value}.  The Open Type of {@code key} is
 616         opentype(K), and the Open Type of {@code value} is
 617         opentype(V).  The index of the {@code TabularType} is the
 618         single item {@code key}.
 619 
 620     For example, the {@code TabularType} for a {@code
 621       Map} might be constructed with code like
 622         this:
 623 
 624     
            
 625 String typeName =
 626     "java.util.Map<java.lang.String, javax.management.ObjectName>";
 627 String[] keyValue =
 628     new String[] {"key", "value"};
 629 OpenType[] openTypes =
 630     new OpenType[] {SimpleType.STRING, SimpleType.OBJECTNAME};
 631 CompositeType rowType =
 632     new CompositeType(typeName, typeName, keyValue, keyValue, openTypes);
 633 TabularType tabularType =
 634     new TabularType(typeName, typeName, rowType, new String[] {"key"});
 635     
 636 
 637     The {@code typeName} here is determined by the 
 638       type name rules detailed below.
 639 
 640     A {@code SortedMap<}K,V{@code >} is mapped in the
 641       same way, but it is only convertible if
 642       K is a class or interface that implements {@link
 643       java.lang.Comparable}.  Thus, a {@code SortedMap}
 644         is convertible, but a
 645         {@code SortedMap} is not.  The conversion of a
 646           {@code SortedMap} instance will fail with an {@code
 647           IllegalArgumentException} if it has a non-null {@link
 648           java.util.SortedMap#comparator() comparator()}.
 649 
 650     A {@code Map<}K,V{@code >} is reconstructed as
 651       a {@code java.util.HashMap<}K,V{@code >};
 652       a {@code SortedMap<}K,V{@code >} as
 653       a {@code java.util.TreeMap<}K,V{@code >}.
 654 
 655     {@code TabularData} is an interface.  The concrete class that is
 656       used to represent a {@code Map<}K,V{@code >} as
 657       Open Data is {@link TabularDataSupport},
 658       or another class implementing {@code
 659       TabularData} that serializes as {@code TabularDataSupport}.

 662     
            Mappings for MXBean interfaces
            
 663 
 664     An MXBean interface, or a type referenced within an MXBean
 665       interface, can reference another MXBean interface, J.
 666       Then opentype(J) is {@code SimpleType.OBJECTNAME} and
 667       opendata(J) is {@code ObjectName}.
 668 
 669     For example, suppose you have two MXBean interfaces like this:
 670 
 671     
            
 672 public interface ProductMXBean {
 673     public ModuleMXBean[] getModules();
 674 }
 675 
 676 public interface ModuleMXBean {
 677     public ProductMXBean getProduct();
 678 }
 679     
 680 
 681     The object implementing the {@code ModuleMXBean} interface
 682       returns from its {@code getProduct} method an object
 683       implementing the {@code ProductMXBean} interface.  The
 684       {@code ModuleMXBean} object and the returned {@code
 685       ProductMXBean} objects must both be registered as MXBeans in the
 686       same MBean Server.
 687 
 688     The method {@code ModuleMXBean.getProduct()} defines an
 689       attribute called {@code Product}.  The Open Type for this
 690       attribute is {@code SimpleType.OBJECTNAME}, and the corresponding
 691       {@code ObjectName} value will be the name under which the
 692       referenced {@code ProductMXBean} is registered in the MBean
 693       Server.
 694 
 695     If you make an MXBean proxy for a {@code ModuleMXBean} and
 696       call its {@code getProduct()} method, the proxy will map the
 697       {@code ObjectName} back into a {@code ProductMXBean} by making
 698       another MXBean proxy.  More formally, when a proxy made with
 699       {@link JMX#newMXBeanProxy(MBeanServerConnection, ObjectName,
 700        Class)
 701       JMX.newMXBeanProxy(mbeanServerConnection, objectNameX,
 702       interfaceX)} needs to map {@code objectNameY} back into {@code
 703       interfaceY}, another MXBean interface, it does so with {@code
 704       JMX.newMXBeanProxy(mbeanServerConnection, objectNameY,
 705       interfaceY)}.  The implementation may return a proxy that was
 706       previously created by a call to {@code JMX.newMXBeanProxy}
 707       with the same parameters, or it may create a new proxy.
 708 
 709     The reverse mapping is illustrated by the following change to the
 710       {@code ModuleMXBean} interface:
 711 
 712     
            
 713 public interface ModuleMXBean {
 714     public ProductMXBean getProduct();
 715     public void setProduct(ProductMXBean c);
 716 }
 717     
 718 
 719     The presence of the {@code setProduct} method now means that the
 720       {@code Product} attribute is read/write.  As before, the value
 721       of this attribute is an {@code ObjectName}.  When the attribute is
 722       set, the {@code ObjectName} must be converted into the
 723       {@code ProductMXBean} object that the {@code setProduct} method
 724       expects.  This object will be an MXBean proxy for the given
 725       {@code ObjectName} in the same MBean Server.
 726 
 727     If you make an MXBean proxy for a {@code ModuleMXBean} and
 728       call its {@code setProduct} method, the proxy will map its
 729       {@code ProductMXBean} argument back into an {@code ObjectName}.
 730       This will only work if the argument is in fact another proxy,
 731       for a {@code ProductMXBean} in the same {@code
 732       MBeanServerConnection}.  The proxy can have been returned from
 733       another proxy (like {@code ModuleMXBean.getProduct()} which
 734       returns a proxy for a {@code ProductMXBean}); or it can have
 735       been created by {@link
 736       JMX#newMXBeanProxy(MBeanServerConnection, ObjectName, Class)
 737       JMX.newMXBeanProxy}; or it can have been created using {@link
 738       java.lang.reflect.Proxy Proxy} with an invocation handler that
 739       is {@link MBeanServerInvocationHandler} or a subclass.
 740 
 741     If the same MXBean were registered under two different
 742       {@code ObjectName}s, a reference to that MXBean from another
 743       MXBean would be ambiguous.  Therefore, if an MXBean object is
 744       already registered in an MBean Server and an attempt is made to
 745       register it in the same MBean Server under another name, the
 746       result is an {@link InstanceAlreadyExistsException}.  Registering
 747       the same MBean object under more than one name is discouraged in
 748       general, notably because it does not work well for MBeans that are
 749       {@link NotificationBroadcaster}s.
 750 
 751     
            Mappings for other types
            
 752 
 753     Given a Java class or interface J that does not match the other
 754       rules in the table above, the MXBean framework will attempt to map
 755       it to a {@link CompositeType} as follows.  The type name of this
 756       {@code CompositeType} is determined by the 
 757       type name rules below.
 758 
 759     The class is examined for getters using the conventions
 760       above.  (Getters must be public
 761       instance methods.)  If there are no getters, or if
 762       any getter has a type that is not convertible, then J is
 763       not convertible.
 764 
 765     If there is at least one getter and every getter has a
 766       convertible type, then opentype(J) is a {@code
 767       CompositeType} with one item for every getter.  If the getter is
 768 
 769     
            
 770       T getName()
 771     
 772 
 773     then the item in the {@code CompositeType} is called {@code name}
 774     and has type opentype(T).  For example, if the item is
 775 
 776     
            
 777       String getOwner()
 778     
 779 
 780     then the item is called {@code owner} and has Open Type {@code
 781     SimpleType.STRING}.  If the getter is
 782 
 783     
            
 784       boolean isName()
 785     
 786 
 787     then the item in the {@code CompositeType} is called {@code name}
 788     and has type {@code SimpleType.BOOLEAN}.
 789 
 790     Notice that the first character (or code point) is converted to
 791       lower case.  This follows the Java Beans convention, which for
 792       historical reasons is different from the Standard MBean
 793       convention.  In a Standard MBean or MXBean interface, a method
 794       {@code getOwner} defines an attribute called {@code Owner}, while
 795       in a Java Bean or mapped {@code CompositeType}, a method {@code
 796       getOwner} defines a property or item called {@code owner}.
 797 
 798     If two methods produce the same item name (for example, {@code
 799       getOwner} and {@code isOwner}, or {@code getOwner} and {@code
 800       getowner}) then the type is not convertible.
 801 
 802     When the Open Type is {@code CompositeType}, the corresponding
 803       mapped Java type (opendata(J)) is {@link
 804       CompositeData}.  The mapping from an instance of J to a
 805       {@code CompositeData} corresponding to the {@code CompositeType}
 806       just described is done as follows.  First, if J
 807       implements the interface {@link CompositeDataView}, then that
 808       interface's {@link CompositeDataView#toCompositeData
 809       toCompositeData} method is called to do the conversion.
 810       Otherwise, the {@code CompositeData} is constructed by calling
 811       the getter for each item and converting it to the corresponding
 812       Open Data type.  Thus, a getter such as
 813 
 814     
            
 815       {@code List getNames()}
 816     
 817 
 818     will have been mapped to an item with name "{@code names}" and
 819       Open Type {@code ArrayType(1, SimpleType.STRING)}.  The conversion
 820       to {@code CompositeData} will call {@code getNames()} and convert
 821       the resultant {@code List} into a {@code String[]} for the
 822         item "{@code names}".
 823 
 824     {@code CompositeData} is an interface.  The concrete class that is
 825       used to represent a type as Open Data is {@link
 826       CompositeDataSupport}, or another class implementing {@code
 827       CompositeData} that serializes as {@code
 828       CompositeDataSupport}.
 829

 831     
            Reconstructing an instance of Java type J from
            
 832       a {@code CompositeData}
 833 
 834     If opendata(J) is {@code CompositeData} for a Java type
 835       J, then either an instance of J can be
 836       reconstructed from a {@code CompositeData}, or J is not
 837       reconstructible.  If any item in the {@code CompositeData} is not
 838       reconstructible, then J is not reconstructible either.
 839 
 840     For any given J, the following rules are consulted to
 841       determine how to reconstruct instances of J from
 842       {@code CompositeData}.  The first applicable rule in the list is
 843       the one that will be used.
 844 
 845     
              
 846 
 847       
            1. If J has a method
              2. 
 848         {@code public static }J {@code from(CompositeData cd)}
              
 849         then that method is called to reconstruct an instance of
 850         J.
 851 
 852       
            2. Otherwise, if J has at least one public
              3. 
 853         constructor with a {@link java.beans.ConstructorProperties
 854         ConstructorProperties} annotation, then one
 855         of those constructors (not necessarily always the same one)
 856         will be called to reconstruct an instance of J.
 857         Every such annotation must list as many strings as the
 858         constructor has parameters; each string must name a property
 859         corresponding to a getter of J; and the type of this
 860         getter must be the same as the corresponding constructor
 861         parameter.  It is not an error for there to be getters that
 862         are not mentioned in the {@code ConstructorProperties} annotation
 863         (these may correspond to information that is not needed to
 864         reconstruct the object).
 865 
 866         An instance of J is reconstructed by calling a
 867         constructor with the appropriate reconstructed items from the
 868         {@code CompositeData}.  The constructor to be called will be
 869         determined at runtime based on the items actually present in
 870         the {@code CompositeData}, given that this {@code
 871         CompositeData} might come from an earlier version of
 872         J where not all the items were present.  A
 873         constructor is applicable if all the properties named
 874         in its {@code ConstructorProperties} annotation are present as items
 875         in the {@code CompositeData}.  If no constructor is
 876         applicable, then the attempt to reconstruct J fails.
 877 
 878         For any possible combination of properties, it must be the
 879         case that either (a) there are no applicable constructors, or
 880         (b) there is exactly one applicable constructor, or (c) one of
 881         the applicable constructors names a proper superset of the
 882         properties named by each other applicable constructor.  (In
 883         other words, there should never be ambiguity over which
 884         constructor to choose.)  If this condition is not true, then
 885         J is not reconstructible.
 886 
 887       
            3. Otherwise, if J has a public no-arg constructor, and
              4. 
 888         for every getter in J with type
 889         T and name N there is a corresponding setter
 890         with the same name and type, then an instance of J is
 891         constructed with the no-arg constructor and the setters are
 892         called with the reconstructed items from the {@code CompositeData}
 893         to restore the values.  For example, if there is a method
              
 894         {@code public List getNames()}
              
 895           then there must also be a method
              
 896           {@code public void setNames(List names)}
              
 897             for this rule to apply.
 898 
 899         If the {@code CompositeData} came from an earlier version of
 900         J, some items might not be present.  In this case,
 901         the corresponding setters will not be called.
 902 
 903       
            4. Otherwise, if J is an interface that has no methods
              5. 
 904         other than getters, an instance of J is constructed
 905         using a {@link java.lang.reflect.Proxy} with a {@link
 906         CompositeDataInvocationHandler} backed by the {@code
 907         CompositeData} being converted.
 908 
 909       
            5. Otherwise, J is not reconstructible.
              6. 
 910     
 911 
 912     Rule 2 is not applicable to subset Profiles of Java SE that do not
 913     include the {@code java.beans} package. When targeting a runtime that does
 914     not include the {@code java.beans} package, and where there is a mismatch
 915     between the compile-time and runtime environment whereby J is
 916     compiled with a public constructor and the {@code ConstructorProperties}
 917     annotation, then J is not reconstructible unless another rule
 918     applies.
 919 
 920     Here are examples showing different ways to code a type {@code
 921       NamedNumber} that consists of an {@code int} and a {@code
 922       String}.  In each case, the {@code CompositeType} looks like this:
 923 
 924     
              
 925       
              
 926 {@link CompositeType}(
 927     "NamedNumber",                      // typeName
 928     "NamedNumber",                      // description
 929     new String[] {"number", "name"},    // itemNames
 930     new String[] {"number", "name"},    // itemDescriptions
 931     new OpenType[] {SimpleType.INTEGER,
 932                     SimpleType.STRING}  // itemTypes
 933 );
 934       
 935     
 936 
 937     
                
 938       
              1. Static {@code from} method:
                2. 
 939 
 940         
                
 941           
                
 942 public class NamedNumber {
 943     public int getNumber() {return number;}
 944     public String getName() {return name;}
 945     private NamedNumber(int number, String name) {
 946         this.number = number;
 947         this.name = name;
 948     }
 949     public static NamedNumber from(CompositeData cd) {
 950         return new NamedNumber((Integer) cd.get("number"),
 951                                (String) cd.get("name"));
 952     }
 953     private final int number;
 954     private final String name;
 955 }
 956           
 957         
 958       
 959 
 960       
              2. Public constructor with @ConstructorProperties annotation:
                3. 
 961

 963           
                
 964 public class NamedNumber {
 965     public int getNumber() {return number;}
 966     public String getName() {return name;}
 967     @ConstructorProperties({"number", "name"})
 968     public NamedNumber(int number, String name) {
 969         this.number = number;
 970         this.name = name;
 971     }
 972     private final int number;
 973     private final String name;
 974 }
 975           
 976         
 977       
 978 
 979       
              3. Setter for every getter:
                4. 
 980 
 981         
                
 982           
                
 983 public class NamedNumber {
 984     public int getNumber() {return number;}
 985     public void setNumber(int number) {this.number = number;}
 986     public String getName() {return name;}
 987     public void setName(String name) {this.name = name;}
 988     public NamedNumber() {}
 989     private int number;
 990     private String name;
 991 }
 992           
 993

              4. Interface with only getters:
                5.

1001     public int getNumber();

1003 }

1009     It is usually better for classes that simply represent a
1010       collection of data to be immutable.  An instance of an
1011       immutable class cannot be changed after it has been constructed.
1012       Notice that {@code CompositeData} itself is immutable.
1013       Immutability has many advantages, notably with regard to
1014       thread-safety and security.  So the approach using setters should
1015       generally be avoided if possible.

                Recursive types
                
1019 
1020     Recursive (self-referential) types cannot be used in MXBean
1021       interfaces.  This is a consequence of the immutability of {@link
1022       CompositeType}.  For example, the following type could not be the
1023       type of an attribute, because it refers to itself:
1024 
1025     
                
1026 public interface Node {
1027     public String getName();
1028     public int getPriority();
1029     public Node getNext();
1030 }
1031 
1032 
1033     It is always possible to rewrite recursive types like this so
1034       they are no longer recursive.  Doing so may require introducing
1035       new types.  For example:
1036 
1037     
                
1038 public interface NodeList {
1039     public List<Node> getNodes();
1040 }
1041 
1042 public interface Node {
1043     public String getName();
1044     public int getPriority();
1045 }

1048     
                MBeanInfo contents for an MXBean
                
1049 
1050     An MXBean is a type of Open MBean.  However, for compatibility
1051       reasons, its {@link MBeanInfo} is not an {@link OpenMBeanInfo}.
1052       In particular, when the type of an attribute, parameter, or
1053       operation return value is a primitive type such as {@code int},
1054       or is {@code void} (for a return type), then the attribute,
1055       parameter, or operation will be represented respectively by an
1056       {@link MBeanAttributeInfo}, {@link MBeanParameterInfo}, or
1057       {@link MBeanOperationInfo} whose {@code getType()} or {@code
1058       getReturnType()} returns the primitive name ("{@code int}" etc).
1059       This is so even though the mapping rules above specify that the
1060       opendata mapping is the wrapped type ({@code Integer}
1061       etc).
1062 
1063     The array of public constructors returned by {@link
1064       MBeanInfo#getConstructors()} for an MXBean that is directly
1065       registered in the MBean Server will contain all of the public
1066       constructors of that MXBean.  If the class of the MXBean is not
1067       public then its constructors are not considered public either.
1068       The list returned for an MXBean that is constructed using the
1069       {@link StandardMBean} class is derived in the same way as for
1070       Standard MBeans.  Regardless of how the MXBean was constructed,
1071       its constructor parameters are not subject to MXBean mapping
1072       rules and do not have a corresponding {@code OpenType}.
1073 
1074     The array of notification types returned by {@link
1075       MBeanInfo#getNotifications()} for an MXBean that is directly
1076       registered in the MBean Server will be empty if the MXBean does
1077       not implement the {@link NotificationBroadcaster} interface.
1078       Otherwise, it will be the result of calling {@link
1079       NotificationBroadcaster#getNotificationInfo()} at the time the MXBean
1080       was registered.  Even if the result of this method changes
1081       subsequently, the result of {@code MBeanInfo.getNotifications()}
1082       will not.  The list returned for an MXBean that is constructed
1083       using the {@link StandardMBean} or {@link StandardEmitterMBean}
1084       class is derived in the same way as for Standard MBeans.
1085 
1086     The {@link Descriptor} for all of the
1087       {@code MBeanAttributeInfo}, {@code MBeanParameterInfo}, and
1088       {@code MBeanOperationInfo} objects contained in the {@code MBeanInfo}
1089       will have a field {@code openType} whose value is the {@link OpenType}
1090       specified by the mapping rules above.  So even when {@code getType()}
1091       is "{@code int}", {@code getDescriptor().getField("openType")} will
1092       be {@link SimpleType#INTEGER}.
1093 
1094     The {@code Descriptor} for each of these objects will also have a
1095       field {@code originalType} that is a string representing the Java type
1096       that appeared in the MXBean interface.  The format of this string
1097       is described in the section Type Names
1098       below.
1099 
1100     The {@code Descriptor} for the {@code MBeanInfo} will have a field
1101       {@code mxbean} whose value is the string "{@code true}".
1102

1104     
                Type Names
                
1105 
1106     Sometimes the unmapped type T of a method parameter or
1107     return value in an MXBean must be represented as a string.  If
1108     T is a non-generic type, this string is the value
1109     returned by {@link Class#getName()}.  Otherwise it is the value of
1110     genericstring(T), defined as follows:
1111 
1112     
                  
1113 
1114       
                • If T is a non-generic non-array type,
                  • 
1115       genericstring(T) is the value returned by {@link
1116       Class#getName()}, for example {@code "int"} or {@code
1117       "java.lang.String"}.
1118 
1119       
                • If T is an array E[],
                  • 
1120       genericstring(T) is genericstring(E) followed
1121       by {@code "[]"}.  For example, genericstring({@code int[]})
1122       is {@code "int[]"}, and genericstring({@code
1123       List[][]}) is {@code
1124       "java.util.List[][]"}.
1125 
1126     
                • Otherwise, T is a parameterized type such as {@code
                  • 
1127     List} and genericstring(T) consists of the
1128     following: the fully-qualified name of the parameterized type as
1129     returned by {@code Class.getName()}; a left angle bracket ({@code
1130     "<"}); genericstring(A) where A is the first
1131     type parameter; if there is a second type parameter B
1132     then {@code ", "} (a comma and a single space) followed by
1133     genericstring(B); a right angle bracket ({@code ">"}).
1134 
1135     
1136 
1137     Note that if a method returns {@code int[]}, this will be
1138       represented by the string {@code "[I"} returned by {@code
1139       Class.getName()}, but if a method returns {@code List},
1140       this will be represented by the string {@code
1141       "java.util.List"}.

                  Exceptions
                  
1144 
1145     A problem with mapping from Java types to
1146       Open types is signaled with an {@link OpenDataException}.  This
1147       can happen when an MXBean interface is being analyzed, for
1148       example if it references a type like {@link java.util.Random
1149       java.util.Random} that has no getters.  Or it can happen when an
1150       instance is being converted (a return value from a method in an
1151       MXBean or a parameter to a method in an MXBean proxy), for
1152       example when converting from {@code SortedSet} to {@code
1153       String[]} if the {@code SortedSet} has a non-null {@code
1154       Comparator}.
1155 
1156     A problem with mapping to Java types from
1157       Open types is signaled with an {@link InvalidObjectException}.
1158       This can happen when an MXBean interface is being analyzed, for
1159       example if it references a type that is not
1160       reconstructible according to the rules above, in a
1161       context where a reconstructible type is required.  Or it can
1162       happen when an instance is being converted (a parameter to a
1163       method in an MXBean or a return value from a method in an MXBean
1164       proxy), for example from a String to an Enum if there is no Enum
1165       constant with that name.
1166 
1167     Depending on the context, the {@code OpenDataException} or
1168       {@code InvalidObjectException} may be wrapped in another
1169       exception such as {@link RuntimeMBeanException} or {@link
1170       UndeclaredThrowableException}.  For every thrown exception,
1171       the condition C will be true: "e is {@code
1172       OpenDataException} or {@code InvalidObjectException} (as
1173       appropriate), or C is true of e.{@link
1174       Throwable#getCause() getCause()}".

1177 */
