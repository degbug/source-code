  57 /**
  58  * An {@link ExecutorService} for running {@link ForkJoinTask}s.
  59  * A {@code ForkJoinPool} provides the entry point for submissions
  60  * from non-{@code ForkJoinTask} clients, as well as management and
  61  * monitoring operations.
  62  *
  63  * A {@code ForkJoinPool} differs from other kinds of {@link
  64  * ExecutorService} mainly by virtue of employing
  65  * work-stealing: all threads in the pool attempt to find and
  66  * execute tasks submitted to the pool and/or created by other active
  67  * tasks (eventually blocking waiting for work if none exist). This
  68  * enables efficient processing when most tasks spawn other subtasks
  69  * (as do most {@code ForkJoinTask}s), as well as when many small
  70  * tasks are submitted to the pool from external clients.  Especially
  71  * when setting asyncMode to true in constructors, {@code
  72  * ForkJoinPool}s may also be appropriate for use with event-style
  73  * tasks that are never joined.
  74  *
  75  * A static {@link #commonPool()} is available and appropriate for
  76  * most applications. The common pool is used by any ForkJoinTask that
  77  * is not explicitly submitted to a specified pool. Using the common
  78  * pool normally reduces resource usage (its threads are slowly
  79  * reclaimed during periods of non-use, and reinstated upon subsequent
  80  * use).
  81  *
  82  * For applications that require separate or custom pools, a {@code
  83  * ForkJoinPool} may be constructed with a given target parallelism
  84  * level; by default, equal to the number of available processors.
  85  * The pool attempts to maintain enough active (or available) threads
  86  * by dynamically adding, suspending, or resuming internal worker
  87  * threads, even if some tasks are stalled waiting to join others.
  88  * However, no such adjustments are guaranteed in the face of blocked
  89  * I/O or other unmanaged synchronization. The nested {@link
  90  * ManagedBlocker} interface enables extension of the kinds of
  91  * synchronization accommodated.
  92  *
  93  * In addition to execution and lifecycle control methods, this
  94  * class provides status check methods (for example
  95  * {@link #getStealCount}) that are intended to aid in developing,
  96  * tuning, and monitoring fork/join applications. Also, method
  97  * {@link #toString} returns indications of pool state in a
  98  * convenient form for informal monitoring.
  99  *
 100  * As is the case with other ExecutorServices, there are three
 101  * main task execution methods summarized in the following table.
 102  * These are designed to be used primarily by clients not already
 103  * engaged in fork/join computations in the current pool.  The main
 104  * forms of these methods accept instances of {@code ForkJoinTask},
 105  * but overloaded forms also allow mixed execution of plain {@code
 106  * Runnable}- or {@code Callable}- based activities as well.  However,
 107  * tasks that are already executing in a pool should normally instead
 108  * use the within-computation forms listed in the table unless using
 109  * async event-style tasks that are not usually joined, in which case
 110  * there is little difference among choice of methods.
 111  *
 112  * 

 113  * Summary of task execution methods
 114  *  
 115  *    
 116  *     Call from non-fork/join clients
 117  *     Call from within fork/join computations
 118  *  
 119  *  
 120  *     Arrange async execution
 121  *     {@link #execute(ForkJoinTask)}
 122  *     {@link ForkJoinTask#fork}
 123  *  
 124  *  
 125  *     Await and obtain result
 126  *     {@link #invoke(ForkJoinTask)}
 127  *     {@link ForkJoinTask#invoke}
 128  *  
 129  *  
 130  *     Arrange exec and obtain Future
 131  *     {@link #submit(ForkJoinTask)}
 132  *     {@link ForkJoinTask#fork} (ForkJoinTasks are Futures)
 133  *  
 134  * 
 135  *
 136  * The common pool is by default constructed with default
 137  * parameters, but these may be controlled by setting three
 138  * {@linkplain System#getProperty system properties}:
 139  * 
    
 140  * 
  • {@code java.util.concurrent.ForkJoinPool.common.parallelism}
    • 
 141  * - the parallelism level, a non-negative integer
 142  * 
  • {@code java.util.concurrent.ForkJoinPool.common.threadFactory}
    • 
 143  * - the class name of a {@link ForkJoinWorkerThreadFactory}
 144  * 
  • {@code java.util.concurrent.ForkJoinPool.common.exceptionHandler}
    • 
 145  * - the class name of a {@link UncaughtExceptionHandler}
 146  * 
 147  * If a {@link SecurityManager} is present and no factory is
 148  * specified, then the default pool uses a factory supplying
 149  * threads that have no {@link Permissions} enabled.
 150  * The system class loader is used to load these classes.
 151  * Upon any error in establishing these settings, default parameters
 152  * are used. It is possible to disable or limit the use of threads in
 153  * the common pool by setting the parallelism property to zero, and/or
 154  * using a factory that may return {@code null}. However doing so may
 155  * cause unjoined tasks to never be executed.
 156  *
 157  * Implementation notes: This implementation restricts the
 158  * maximum number of running threads to 32767. Attempts to create
 159  * pools with greater than the maximum number result in
 160  * {@code IllegalArgumentException}.
 161  *
 162  * This implementation rejects submitted tasks (that is, by throwing
 163  * {@link RejectedExecutionException}) only when the pool is shut down
 164  * or internal resources have been exhausted.

 172     /*
 173      * Implementation Overview
 174      *
 175      * This class and its nested classes provide the main
 176      * functionality and control for a set of worker threads:
 177      * Submissions from non-FJ threads enter into submission queues.
 178      * Workers take these tasks and typically split them into subtasks
 179      * that may be stolen by other workers.  Preference rules give
 180      * first priority to processing tasks from their own queues (LIFO
 181      * or FIFO, depending on mode), then to randomized FIFO steals of
 182      * tasks in other queues.  This framework began as vehicle for
 183      * supporting tree-structured parallelism using work-stealing.
 184      * Over time, its scalability advantages led to extensions and
 185      * changes to better support more diverse usage contexts.  Because
 186      * most internal methods and nested classes are interrelated,
 187      * their main rationale and descriptions are presented here;
 188      * individual methods and nested classes contain only brief
 189      * comments about details.
 190      *
 191      * WorkQueues
 192      * ==========
 193      *
 194      * Most operations occur within work-stealing queues (in nested
 195      * class WorkQueue).  These are special forms of Deques that
 196      * support only three of the four possible end-operations -- push,
 197      * pop, and poll (aka steal), under the further constraints that
 198      * push and pop are called only from the owning thread (or, as
 199      * extended here, under a lock), while poll may be called from
 200      * other threads.  (If you are unfamiliar with them, you probably
 201      * want to read Herlihy and Shavit's book "The Art of
 202      * Multiprocessor programming", chapter 16 describing these in
 203      * more detail before proceeding.)  The main work-stealing queue
 204      * design is roughly similar to those in the papers "Dynamic
 205      * Circular Work-Stealing Deque" by Chase and Lev, SPAA 2005
 206      * (http://research.sun.com/scalable/pubs/index.html) and
 207      * "Idempotent work stealing" by Michael, Saraswat, and Vechev,
 208      * PPoPP 2009 (http://portal.acm.org/citation.cfm?id=1504186).
 209      * The main differences ultimately stem from GC requirements that
 210      * we null out taken slots as soon as we can, to maintain as small
 211      * a footprint as possible even in programs generating huge
 212      * numbers of tasks. To accomplish this, we shift the CAS
 213      * arbitrating pop vs poll (steal) from being on the indices
 214      * ("base" and "top") to the slots themselves.
 215      *
 216      * Adding tasks then takes the form of a classic array push(task):
 217      *    q.array[q.top] = task; ++q.top;
 218      *
 219      * (The actual code needs to null-check and size-check the array,
 220      * properly fence the accesses, and possibly signal waiting
 221      * workers to start scanning -- see below.)  Both a successful pop
 222      * and poll mainly entail a CAS of a slot from non-null to null.
 223      *
 224      * The pop operation (always performed by owner) is:
 225      *   if ((base != top) and
 226      *        (the task at top slot is not null) and
 227      *        (CAS slot to null))
 228      *           decrement top and return task;
 229      *
 230      * And the poll operation (usually by a stealer) is
 231      *    if ((base != top) and
 232      *        (the task at base slot is not null) and
 233      *        (base has not changed) and
 234      *        (CAS slot to null))
 235      *           increment base and return task;
 236      *
 237      * Because we rely on CASes of references, we do not need tag bits
 238      * on base or top.  They are simple ints as used in any circular
 239      * array-based queue (see for example ArrayDeque).  Updates to the
 240      * indices guarantee that top == base means the queue is empty,
 241      * but otherwise may err on the side of possibly making the queue
 242      * appear nonempty when a push, pop, or poll have not fully
 243      * committed. (Method isEmpty() checks the case of a partially
 244      * completed removal of the last element.)  Because of this, the
 245      * poll operation, considered individually, is not wait-free. One
 246      * thief cannot successfully continue until another in-progress
 247      * one (or, if previously empty, a push) completes.  However, in
 248      * the aggregate, we ensure at least probabilistic
 249      * non-blockingness.  If an attempted steal fails, a thief always
 250      * chooses a different random victim target to try next. So, in
 251      * order for one thief to progress, it suffices for any
 252      * in-progress poll or new push on any empty queue to
 253      * complete. (This is why we normally use method pollAt and its
 254      * variants that try once at the apparent base index, else
 255      * consider alternative actions, rather than method poll, which
 256      * retries.)
 257      *
 258      * This approach also enables support of a user mode in which
 259      * local task processing is in FIFO, not LIFO order, simply by
 260      * using poll rather than pop.  This can be useful in
 261      * message-passing frameworks in which tasks are never joined.
 262      * However neither mode considers affinities, loads, cache
 263      * localities, etc, so rarely provide the best possible
 264      * performance on a given machine, but portably provide good
 265      * throughput by averaging over these factors.  Further, even if
 266      * we did try to use such information, we do not usually have a
 267      * basis for exploiting it.  For example, some sets of tasks
 268      * profit from cache affinities, but others are harmed by cache
 269      * pollution effects. Additionally, even though it requires
 270      * scanning, long-term throughput is often best using random
 271      * selection rather than directed selection policies, so cheap
 272      * randomization of sufficient quality is used whenever
 273      * applicable.  Various Marsaglia XorShifts (some with different
 274      * shift constants) are inlined at use points.
 275      *
 276      * WorkQueues are also used in a similar way for tasks submitted
 277      * to the pool. We cannot mix these tasks in the same queues used
 278      * by workers. Instead, we randomly associate submission queues
 279      * with submitting threads, using a form of hashing.  The
 280      * ThreadLocalRandom probe value serves as a hash code for
 281      * choosing existing queues, and may be randomly repositioned upon
 282      * contention with other submitters.  In essence, submitters act
 283      * like workers except that they are restricted to executing local
 284      * tasks that they submitted (or in the case of CountedCompleters,
 285      * others with the same root task).  Insertion of tasks in shared
 286      * mode requires a lock (mainly to protect in the case of
 287      * resizing) but we use only a simple spinlock (using field
 288      * qlock), because submitters encountering a busy queue move on to
 289      * try or create other queues -- they block only when creating and
 290      * registering new queues. Additionally, "qlock" saturates to an
 291      * unlockable value (-1) at shutdown. Unlocking still can be and
 292      * is performed by cheaper ordered writes of "qlock" in successful
 293      * cases, but uses CAS in unsuccessful cases.
 294      *
 295      * Management
 296      * ==========
 297      *
 298      * The main throughput advantages of work-stealing stem from
 299      * decentralized control -- workers mostly take tasks from
 300      * themselves or each other, at rates that can exceed a billion
 301      * per second.  The pool itself creates, activates (enables
 302      * scanning for and running tasks), deactivates, blocks, and
 303      * terminates threads, all with minimal central information.
 304      * There are only a few properties that we can globally track or
 305      * maintain, so we pack them into a small number of variables,
 306      * often maintaining atomicity without blocking or locking.
 307      * Nearly all essentially atomic control state is held in two
 308      * volatile variables that are by far most often read (not
 309      * written) as status and consistency checks. (Also, field
 310      * "config" holds unchanging configuration state.)
 311      *
 312      * Field "ctl" contains 64 bits holding information needed to
 313      * atomically decide to add, inactivate, enqueue (on an event
 314      * queue), dequeue, and/or re-activate workers.  To enable this
 315      * packing, we restrict maximum parallelism to (1<<15)-1 (which is
 316      * far in excess of normal operating range) to allow ids, counts,
 317      * and their negations (used for thresholding) to fit into 16bit
 318      * subfields.
 319      *
 320      * Field "runState" holds lockable state bits (STARTED, STOP, etc)
 321      * also protecting updates to the workQueues array.  When used as
 322      * a lock, it is normally held only for a few instructions (the
 323      * only exceptions are one-time array initialization and uncommon
 324      * resizing), so is nearly always available after at most a brief
 325      * spin. But to be extra-cautious, after spinning, method
 326      * awaitRunStateLock (called only if an initial CAS fails), uses a
 327      * wait/notify mechanics on a builtin monitor to block when
 328      * (rarely) needed. This would be a terrible idea for a highly
 329      * contended lock, but most pools run without the lock ever
 330      * contending after the spin limit, so this works fine as a more
 331      * conservative alternative. Because we don't otherwise have an
 332      * internal Object to use as a monitor, the "stealCounter" (an
 333      * AtomicLong) is used when available (it too must be lazily
 334      * initialized; see externalSubmit).
 335      *
 336      * Usages of "runState" vs "ctl" interact in only one case:
 337      * deciding to add a worker thread (see tryAddWorker), in which
 338      * case the ctl CAS is performed while the lock is held.
 339      *
 340      * Recording WorkQueues.  WorkQueues are recorded in the
 341      * "workQueues" array. The array is created upon first use (see
 342      * externalSubmit) and expanded if necessary.  Updates to the
 343      * array while recording new workers and unrecording terminated
 344      * ones are protected from each other by the runState lock, but
 345      * the array is otherwise concurrently readable, and accessed
 346      * directly. We also ensure that reads of the array reference
 347      * itself never become too stale. To simplify index-based
 348      * operations, the array size is always a power of two, and all
 349      * readers must tolerate null slots. Worker queues are at odd
 350      * indices. Shared (submission) queues are at even indices, up to
 351      * a maximum of 64 slots, to limit growth even if array needs to
 352      * expand to add more workers. Grouping them together in this way
 353      * simplifies and speeds up task scanning.
 354      *
 355      * All worker thread creation is on-demand, triggered by task
 356      * submissions, replacement of terminated workers, and/or
 357      * compensation for blocked workers. However, all other support
 358      * code is set up to work with other policies.  To ensure that we
 359      * do not hold on to worker references that would prevent GC, All
 360      * accesses to workQueues are via indices into the workQueues
 361      * array (which is one source of some of the messy code
 362      * constructions here). In essence, the workQueues array serves as
 363      * a weak reference mechanism. Thus for example the stack top
 364      * subfield of ctl stores indices, not references.
 365      *
 366      * Queuing Idle Workers. Unlike HPC work-stealing frameworks, we
 367      * cannot let workers spin indefinitely scanning for tasks when
 368      * none can be found immediately, and we cannot start/resume
 369      * workers unless there appear to be tasks available.  On the
 370      * other hand, we must quickly prod them into action when new
 371      * tasks are submitted or generated. In many usages, ramp-up time
 372      * to activate workers is the main limiting factor in overall
 373      * performance, which is compounded at program start-up by JIT
 374      * compilation and allocation. So we streamline this as much as
 375      * possible.
 376      *
 377      * The "ctl" field atomically maintains active and total worker
 378      * counts as well as a queue to place waiting threads so they can
 379      * be located for signalling. Active counts also play the role of
 380      * quiescence indicators, so are decremented when workers believe
 381      * that there are no more tasks to execute. The "queue" is
 382      * actually a form of Treiber stack.  A stack is ideal for
 383      * activating threads in most-recently used order. This improves
 384      * performance and locality, outweighing the disadvantages of
 385      * being prone to contention and inability to release a worker
 386      * unless it is topmost on stack.  We park/unpark workers after
 387      * pushing on the idle worker stack (represented by the lower
 388      * 32bit subfield of ctl) when they cannot find work.  The top
 389      * stack state holds the value of the "scanState" field of the
 390      * worker: its index and status, plus a version counter that, in
 391      * addition to the count subfields (also serving as version
 392      * stamps) provide protection against Treiber stack ABA effects.
 393      *
 394      * Field scanState is used by both workers and the pool to manage
 395      * and track whether a worker is INACTIVE (possibly blocked
 396      * waiting for a signal), or SCANNING for tasks (when neither hold
 397      * it is busy running tasks).  When a worker is inactivated, its
 398      * scanState field is set, and is prevented from executing tasks,
 399      * even though it must scan once for them to avoid queuing
 400      * races. Note that scanState updates lag queue CAS releases so
 401      * usage requires care. When queued, the lower 16 bits of
 402      * scanState must hold its pool index. So we place the index there
 403      * upon initialization (see registerWorker) and otherwise keep it
 404      * there or restore it when necessary.
 405      *
 406      * Memory ordering.  See "Correct and Efficient Work-Stealing for
 407      * Weak Memory Models" by Le, Pop, Cohen, and Nardelli, PPoPP 2013
 408      * (http://www.di.ens.fr/~zappa/readings/ppopp13.pdf) for an
 409      * analysis of memory ordering requirements in work-stealing
 410      * algorithms similar to the one used here.  We usually need
 411      * stronger than minimal ordering because we must sometimes signal
 412      * workers, requiring Dekker-like full-fences to avoid lost
 413      * signals.  Arranging for enough ordering without expensive
 414      * over-fencing requires tradeoffs among the supported means of
 415      * expressing access constraints. The most central operations,
 416      * taking from queues and updating ctl state, require full-fence
 417      * CAS.  Array slots are read using the emulation of volatiles
 418      * provided by Unsafe.  Access from other threads to WorkQueue
 419      * base, top, and array requires a volatile load of the first of
 420      * any of these read.  We use the convention of declaring the
 421      * "base" index volatile, and always read it before other fields.
 422      * The owner thread must ensure ordered updates, so writes use
 423      * ordered intrinsics unless they can piggyback on those for other
 424      * writes.  Similar conventions and rationales hold for other
 425      * WorkQueue fields (such as "currentSteal") that are only written
 426      * by owners but observed by others.
 427      *
 428      * Creating workers. To create a worker, we pre-increment total
 429      * count (serving as a reservation), and attempt to construct a
 430      * ForkJoinWorkerThread via its factory. Upon construction, the
 431      * new thread invokes registerWorker, where it constructs a
 432      * WorkQueue and is assigned an index in the workQueues array
 433      * (expanding the array if necessary). The thread is then
 434      * started. Upon any exception across these steps, or null return
 435      * from factory, deregisterWorker adjusts counts and records
 436      * accordingly.  If a null return, the pool continues running with
 437      * fewer than the target number workers. If exceptional, the
 438      * exception is propagated, generally to some external caller.
 439      * Worker index assignment avoids the bias in scanning that would
 440      * occur if entries were sequentially packed starting at the front
 441      * of the workQueues array. We treat the array as a simple
 442      * power-of-two hash table, expanding as needed. The seedIndex
 443      * increment ensures no collisions until a resize is needed or a
 444      * worker is deregistered and replaced, and thereafter keeps
 445      * probability of collision low. We cannot use
 446      * ThreadLocalRandom.getProbe() for similar purposes here because
 447      * the thread has not started yet, but do so for creating
 448      * submission queues for existing external threads.
 449      *
 450      * Deactivation and waiting. Queuing encounters several intrinsic
 451      * races; most notably that a task-producing thread can miss
 452      * seeing (and signalling) another thread that gave up looking for
 453      * work but has not yet entered the wait queue.  When a worker
 454      * cannot find a task to steal, it deactivates and enqueues. Very
 455      * often, the lack of tasks is transient due to GC or OS
 456      * scheduling. To reduce false-alarm deactivation, scanners
 457      * compute checksums of queue states during sweeps.  (The
 458      * stability checks used here and elsewhere are probabilistic
 459      * variants of snapshot techniques -- see Herlihy & Shavit.)
 460      * Workers give up and try to deactivate only after the sum is
 461      * stable across scans. Further, to avoid missed signals, they
 462      * repeat this scanning process after successful enqueuing until
 463      * again stable.  In this state, the worker cannot take/run a task
 464      * it sees until it is released from the queue, so the worker
 465      * itself eventually tries to release itself or any successor (see
 466      * tryRelease).  Otherwise, upon an empty scan, a deactivated
 467      * worker uses an adaptive local spin construction (see awaitWork)
 468      * before blocking (via park). Note the unusual conventions about
 469      * Thread.interrupts surrounding parking and other blocking:
 470      * Because interrupts are used solely to alert threads to check
 471      * termination, which is checked anyway upon blocking, we clear
 472      * status (using Thread.interrupted) before any call to park, so
 473      * that park does not immediately return due to status being set
 474      * via some other unrelated call to interrupt in user code.
 475      *
 476      * Signalling and activation.  Workers are created or activated
 477      * only when there appears to be at least one task they might be
 478      * able to find and execute.  Upon push (either by a worker or an
 479      * external submission) to a previously (possibly) empty queue,
 480      * workers are signalled if idle, or created if fewer exist than
 481      * the given parallelism level.  These primary signals are
 482      * buttressed by others whenever other threads remove a task from
 483      * a queue and notice that there are other tasks there as well.
 484      * On most platforms, signalling (unpark) overhead time is
 485      * noticeably long, and the time between signalling a thread and
 486      * it actually making progress can be very noticeably long, so it
 487      * is worth offloading these delays from critical paths as much as
 488      * possible. Also, because inactive workers are often rescanning
 489      * or spinning rather than blocking, we set and clear the "parker"
 490      * field of WorkQueues to reduce unnecessary calls to unpark.
 491      * (This requires a secondary recheck to avoid missed signals.)
 492      *
 493      * Trimming workers. To release resources after periods of lack of
 494      * use, a worker starting to wait when the pool is quiescent will
 495      * time out and terminate (see awaitWork) if the pool has remained
 496      * quiescent for period IDLE_TIMEOUT, increasing the period as the
 497      * number of threads decreases, eventually removing all workers.
 498      * Also, when more than two spare threads exist, excess threads
 499      * are immediately terminated at the next quiescent point.
 500      * (Padding by two avoids hysteresis.)
 501      *
 502      * Shutdown and Termination. A call to shutdownNow invokes
 503      * tryTerminate to atomically set a runState bit. The calling
 504      * thread, as well as every other worker thereafter terminating,
 505      * helps terminate others by setting their (qlock) status,
 506      * cancelling their unprocessed tasks, and waking them up, doing
 507      * so repeatedly until stable (but with a loop bounded by the
 508      * number of workers).  Calls to non-abrupt shutdown() preface
 509      * this by checking whether termination should commence. This
 510      * relies primarily on the active count bits of "ctl" maintaining
 511      * consensus -- tryTerminate is called from awaitWork whenever
 512      * quiescent. However, external submitters do not take part in
 513      * this consensus.  So, tryTerminate sweeps through queues (until
 514      * stable) to ensure lack of in-flight submissions and workers
 515      * about to process them before triggering the "STOP" phase of
 516      * termination. (Note: there is an intrinsic conflict if
 517      * helpQuiescePool is called when shutdown is enabled. Both wait
 518      * for quiescence, but tryTerminate is biased to not trigger until
 519      * helpQuiescePool completes.)
 520      *
 521      *
 522      * Joining Tasks
 523      * =============
 524      *
 525      * Any of several actions may be taken when one worker is waiting
 526      * to join a task stolen (or always held) by another.  Because we
 527      * are multiplexing many tasks on to a pool of workers, we can't
 528      * just let them block (as in Thread.join).  We also cannot just
 529      * reassign the joiner's run-time stack with another and replace
 530      * it later, which would be a form of "continuation", that even if
 531      * possible is not necessarily a good idea since we may need both
 532      * an unblocked task and its continuation to progress.  Instead we
 533      * combine two tactics:
 534      *
 535      *   Helping: Arranging for the joiner to execute some task that it
 536      *      would be running if the steal had not occurred.
 537      *
 538      *   Compensating: Unless there are already enough live threads,
 539      *      method tryCompensate() may create or re-activate a spare
 540      *      thread to compensate for blocked joiners until they unblock.
 541      *
 542      * A third form (implemented in tryRemoveAndExec) amounts to
 543      * helping a hypothetical compensator: If we can readily tell that
 544      * a possible action of a compensator is to steal and execute the
 545      * task being joined, the joining thread can do so directly,
 546      * without the need for a compensation thread (although at the
 547      * expense of larger run-time stacks, but the tradeoff is
 548      * typically worthwhile).
 549      *
 550      * The ManagedBlocker extension API can't use helping so relies
 551      * only on compensation in method awaitBlocker.
 552      *
 553      * The algorithm in helpStealer entails a form of "linear
 554      * helping".  Each worker records (in field currentSteal) the most
 555      * recent task it stole from some other worker (or a submission).
 556      * It also records (in field currentJoin) the task it is currently
 557      * actively joining. Method helpStealer uses these markers to try
 558      * to find a worker to help (i.e., steal back a task from and
 559      * execute it) that could hasten completion of the actively joined
 560      * task.  Thus, the joiner executes a task that would be on its
 561      * own local deque had the to-be-joined task not been stolen. This
 562      * is a conservative variant of the approach described in Wagner &
 563      * Calder "Leapfrogging: a portable technique for implementing
 564      * efficient futures" SIGPLAN Notices, 1993
 565      * (http://portal.acm.org/citation.cfm?id=155354). It differs in
 566      * that: (1) We only maintain dependency links across workers upon
 567      * steals, rather than use per-task bookkeeping.  This sometimes
 568      * requires a linear scan of workQueues array to locate stealers,
 569      * but often doesn't because stealers leave hints (that may become
 570      * stale/wrong) of where to locate them.  It is only a hint
 571      * because a worker might have had multiple steals and the hint
 572      * records only one of them (usually the most current).  Hinting
 573      * isolates cost to when it is needed, rather than adding to
 574      * per-task overhead.  (2) It is "shallow", ignoring nesting and
 575      * potentially cyclic mutual steals.  (3) It is intentionally
 576      * racy: field currentJoin is updated only while actively joining,
 577      * which means that we miss links in the chain during long-lived
 578      * tasks, GC stalls etc (which is OK since blocking in such cases
 579      * is usually a good idea).  (4) We bound the number of attempts
 580      * to find work using checksums and fall back to suspending the
 581      * worker and if necessary replacing it with another.
 582      *
 583      * Helping actions for CountedCompleters do not require tracking
 584      * currentJoins: Method helpComplete takes and executes any task
 585      * with the same root as the task being waited on (preferring
 586      * local pops to non-local polls). However, this still entails
 587      * some traversal of completer chains, so is less efficient than
 588      * using CountedCompleters without explicit joins.
 589      *
 590      * Compensation does not aim to keep exactly the target
 591      * parallelism number of unblocked threads running at any given
 592      * time. Some previous versions of this class employed immediate
 593      * compensations for any blocked join. However, in practice, the
 594      * vast majority of blockages are transient byproducts of GC and
 595      * other JVM or OS activities that are made worse by replacement.
 596      * Currently, compensation is attempted only after validating that
 597      * all purportedly active threads are processing tasks by checking
 598      * field WorkQueue.scanState, which eliminates most false
 599      * positives.  Also, compensation is bypassed (tolerating fewer
 600      * threads) in the most common case in which it is rarely
 601      * beneficial: when a worker with an empty queue (thus no
 602      * continuation tasks) blocks on a join and there still remain
 603      * enough threads to ensure liveness.
 604      *
 605      * The compensation mechanism may be bounded.  Bounds for the
 606      * commonPool (see commonMaxSpares) better enable JVMs to cope
 607      * with programming errors and abuse before running out of
 608      * resources to do so. In other cases, users may supply factories
 609      * that limit thread construction. The effects of bounding in this
 610      * pool (like all others) is imprecise.  Total worker counts are
 611      * decremented when threads deregister, not when they exit and
 612      * resources are reclaimed by the JVM and OS. So the number of
 613      * simultaneously live threads may transiently exceed bounds.
 614      *
 615      * Common Pool
 616      * ===========
 617      *
 618      * The static common pool always exists after static
 619      * initialization.  Since it (or any other created pool) need
 620      * never be used, we minimize initial construction overhead and
 621      * footprint to the setup of about a dozen fields, with no nested
 622      * allocation. Most bootstrapping occurs within method
 623      * externalSubmit during the first submission to the pool.
 624      *
 625      * When external threads submit to the common pool, they can
 626      * perform subtask processing (see externalHelpComplete and
 627      * related methods) upon joins.  This caller-helps policy makes it
 628      * sensible to set common pool parallelism level to one (or more)
 629      * less than the total number of available cores, or even zero for
 630      * pure caller-runs.  We do not need to record whether external
 631      * submissions are to the common pool -- if not, external help
 632      * methods return quickly. These submitters would otherwise be
 633      * blocked waiting for completion, so the extra effort (with
 634      * liberally sprinkled task status checks) in inapplicable cases
 635      * amounts to an odd form of limited spin-wait before blocking in
 636      * ForkJoinTask.join.
 637      *
 638      * As a more appropriate default in managed environments, unless
 639      * overridden by system properties, we use workers of subclass
 640      * InnocuousForkJoinWorkerThread when there is a SecurityManager
 641      * present. These workers have no permissions set, do not belong
 642      * to any user-defined ThreadGroup, and erase all ThreadLocals
 643      * after executing any top-level task (see WorkQueue.runTask).
 644      * The associated mechanics (mainly in ForkJoinWorkerThread) may
 645      * be JVM-dependent and must access particular Thread class fields
 646      * to achieve this effect.

 648      * Style notes
 649      * ===========
 650      *
 651      * Memory ordering relies mainly on Unsafe intrinsics that carry
 652      * the further responsibility of explicitly performing null- and
 653      * bounds- checks otherwise carried out implicitly by JVMs.  This
 654      * can be awkward and ugly, but also reflects the need to control
 655      * outcomes across the unusual cases that arise in very racy code
 656      * with very few invariants. So these explicit checks would exist
 657      * in some form anyway.  All fields are read into locals before
 658      * use, and null-checked if they are references.  This is usually
 659      * done in a "C"-like style of listing declarations at the heads
 660      * of methods or blocks, and using inline assignments on first
 661      * encounter.  Array bounds-checks are usually performed by
 662      * masking with array.length-1, which relies on the invariant that
 663      * these arrays are created with positive lengths, which is itself
 664      * paranoically checked. Nearly all explicit checks lead to
 665      * bypass/return, not exception throws, because they may
 666      * legitimately arise due to cancellation/revocation during
 667      * shutdown.
 668      *
 669      * There is a lot of representation-level coupling among classes
 670      * ForkJoinPool, ForkJoinWorkerThread, and ForkJoinTask.  The
 671      * fields of WorkQueue maintain data structures managed by
 672      * ForkJoinPool, so are directly accessed.  There is little point
 673      * trying to reduce this, since any associated future changes in
 674      * representations will need to be accompanied by algorithmic
 675      * changes anyway. Several methods intrinsically sprawl because
 676      * they must accumulate sets of consistent reads of fields held in
 677      * local variables.  There are also other coding oddities
 678      * (including several unnecessary-looking hoisted null checks)
 679      * that help some methods perform reasonably even when interpreted
 680      * (not compiled).

 688      * (6) Callbacks and other support for ForkJoinTask methods

 852         /**
 853          * Pushes a task. Call only by owner in unshared queues.  (The
 854          * shared-queue version is embedded in method externalPush.)
 855          *
 856          * @param task the task. Caller must ensure non-null.
 857          * @throws RejectedExecutionException if array cannot be resized
 858          */
 859         final void push(ForkJoinTask<?> task) {
 860             ForkJoinTask<?>[] a; ForkJoinPool p;
 861             int b = base, s = top, n;
 862             if ((a = array) != null) {    // ignore if queue removed
 863                 int m = a.length - 1;     // fenced write for task visibility
 864                 U.putOrderedObject(a, ((m & s) << ASHIFT) + ABASE, task);
 865                 U.putOrderedInt(this, QTOP, s + 1);
 866                 if ((n = s - b) <= 1) {
 867                     if ((p = pool) != null)
 868                         p.signalWork(p.workQueues, this);
 869                 }
 870                 else if (n >= m)

 872             }
 873         }

 875         /**
 876          * Initializes or doubles the capacity of array. Call either
 877          * by owner or with lock held -- it is OK for base, but not
 878          * top, to move while resizings are in progress.
 879          */
 880         final ForkJoinTask<?>[] growArray() {
 881             ForkJoinTask<?>[] oldA = array;
 882             int size = oldA != null ? oldA.length << 1 : INITIAL_QUEUE_CAPACITY;
 883             if (size > MAXIMUM_QUEUE_CAPACITY)
 884                 throw new RejectedExecutionException("Queue capacity exceeded");
 885             int oldMask, t, b;
 886             ForkJoinTask<?>[] a = array = new ForkJoinTask<?>[size];
 887             if (oldA != null && (oldMask = oldA.length - 1) >= 0 &&
 888                 (t = top) - (b = base) > 0) {
 889                 int mask = size - 1;
 890                 do { // emulate poll from old array, push to new array
 891                     ForkJoinTask<?> x;
 892                     int oldj = ((b & oldMask) << ASHIFT) + ABASE;
 893                     int j    = ((b &    mask) << ASHIFT) + ABASE;
 894                     x = (ForkJoinTask<?>)U.getObjectVolatile(oldA, oldj);
 895                     if (x != null &&
 896                         U.compareAndSwapObject(oldA, oldj, x, null))
 897                         U.putObjectVolatile(a, j, x);
 898                 } while (++b != t);
 899             }
 900             return a;
 901         }

 905          * by owner in unshared queues.

 908             ForkJoinTask<?>[] a; ForkJoinTask<?> t; int m;
 909             if ((a = array) != null && (m = a.length - 1) >= 0) {
 910                 for (int s; (s = top - 1) - base >= 0;) {
 911                     long j = ((m & s) << ASHIFT) + ABASE;
 912                     if ((t = (ForkJoinTask<?>)U.getObject(a, j)) == null)
 913                         break;
 914                     if (U.compareAndSwapObject(a, j, t, null)) {
 915                         U.putOrderedInt(this, QTOP, s);
 916                         return t;
 917                     }
 918                 }
 919             }
 920             return null;
 921         }

 933                     base == b && U.compareAndSwapObject(a, j, t, null)) {

 945             ForkJoinTask<?>[] a; int b; ForkJoinTask<?> t;
 946             while ((b = base) - top < 0 && (a = array) != null) {
 947                 int j = (((a.length - 1) & b) << ASHIFT) + ABASE;
 948                 t = (ForkJoinTask<?>)U.getObjectVolatile(a, j);
 949                 if (base == b) {
 950                     if (t != null) {
 951                         if (U.compareAndSwapObject(a, j, t, null)) {
 952                             base = b + 1;
 953                             return t;
 954                         }
 955                     }
 956                     else if (b + 1 == top) // now empty
 957                         break;
 958                 }
 959             }
 960             return null;
 961         }

1050         /**

1053         final void runTask(ForkJoinTask<?> task) {
1054             if (task != null) {

1343     /*
1344      * Bits and masks for field ctl, packed with 4 16 bit subfields:
1345      * AC: Number of active running workers minus target parallelism
1346      * TC: Number of total workers minus target parallelism
1347      * SS: version count and status of top waiting thread
1348      * ID: poolIndex of top of Treiber stack of waiters
1349      *
1350      * When convenient, we can extract the lower 32 stack top bits
1351      * (including version bits) as sp=(int)ctl.  The offsets of counts
1352      * by the target parallelism and the positionings of fields makes
1353      * it possible to perform the most common checks via sign tests of
1354      * fields: When ac is negative, there are not enough active
1355      * workers, when tc is negative, there are not enough total
1356      * workers.  When sp is non-zero, there are waiting workers.  To
1357      * deal with possibly negative fields, we use casts in and out of
1358      * "short" and/or signed shifts to maintain signedness.

1361      * using getAndAddLong of AC_UNIT, rather than CAS, when returning
1362      * from a blocked join.  Other updates entail multiple subfields
1363      * and masking, requiring CAS.

1410     /**
1411      * Spins and/or blocks until runstate lock is available.  See
1412      * above for explanation.
1413      */
1414     private int awaitRunStateLock() {
1415         Object lock;
1416         boolean wasInterrupted = false;
1417         for (int spins = SPINS, r = 0, rs, ns;;) {
1418             if (((rs = runState) & RSLOCK) == 0) {
1419                 if (U.compareAndSwapInt(this, RUNSTATE, rs, ns = rs | RSLOCK)) {
1420                     if (wasInterrupted) {
1421                         try {
1422                             Thread.currentThread().interrupt();
1423                         } catch (SecurityException ignore) {
1424                         }
1425                     }
1426                     return ns;
1427                 }
1428             }
1429             else if (r == 0)
1430                 r = ThreadLocalRandom.nextSecondarySeed();
1431             else if (spins > 0) {
1432                 r ^= r << 6; r ^= r >>> 21; r ^= r << 7; // xorshift
1433                 if (r >= 0)
1434                     --spins;
1435             }
1436             else if ((rs & STARTED) == 0 || (lock = stealCounter) == null)
1437                 Thread.yield();   // initialization race
1438             else if (U.compareAndSwapInt(this, RUNSTATE, rs, rs | RSIGNAL)) {
1439                 synchronized (lock) {
1440                     if ((runState & RSIGNAL) != 0) {
1441                         try {
1442                             lock.wait();
1443                         } catch (InterruptedException ie) {
1444                             if (!(Thread.currentThread() instanceof
1445                                   ForkJoinWorkerThread))
1446                                 wasInterrupted = true;
1447                         }
1448                     }
1449                     else
1450                         lock.notifyAll();
1451                 }
1452             }
1453         }
1454     }

1497      * Tries to add one worker, incrementing ctl counts before doing
1498      * so, relying on createWorker to back out on failure.
1499      *
1500      * @param c incoming ctl value, with total count negative and no
1501      * idle workers.  On CAS failure, c is refreshed and retried if
1502      * this holds (otherwise, a new worker is not needed).
1503      */

1505         boolean add = false;
1506         do {
1507             long nc = ((AC_MASK & (c + AC_UNIT)) |
1508                        (TC_MASK & (c + TC_UNIT)));

1511                 if ((stop = (rs = lockRunState()) & STOP) == 0)
1512                     add = U.compareAndSwapLong(this, CTL, c, nc);
1513                 unlockRunState(rs, rs & ~RSLOCK);
1514                 if (stop != 0)
1515                     break;
1516                 if (add) {
1517                     createWorker();
1518                     break;
1519                 }
1520             }

1533         wt.setDaemon(true);                           // configure thread
1534         if ((handler = ueh) != null)
1535             wt.setUncaughtExceptionHandler(handler);

1538         int mode = config & MODE_MASK;

1541             WorkQueue[] ws; int n;                    // skip if no array
1542             if ((ws = workQueues) != null && (n = ws.length) > 0) {
1543                 int s = indexSeed += SEED_INCREMENT;  // unlikely to collide
1544                 int m = n - 1;

1548                     int step = (n <= 4) ? 2 : ((n >>> 1) & EVENMASK) + 2;

1558                 w.config = i | mode;
1559                 w.scanState = i;                      // publication fence
1560                 ws[i] = w;

1571      * to construct or start a worker.  Removes record of worker from
1572      * array, and adjusts counts. If pool is shutting down, tries to
1573      * complete termination.
1574      *
1575      * @param wt the worker thread, or null if construction failed
1576      * @param ex the exception causing failure, or null if none
1577      */
1578     final void deregisterWorker(ForkJoinWorkerThread wt, Throwable ex) {
1579         WorkQueue w = null;
1580         if (wt != null && (w = wt.workQueue) != null) {
1581             WorkQueue[] ws;                           // remove index from array
1582             int idx = w.config & SMASK;
1583             int rs = lockRunState();
1584             if ((ws = workQueues) != null && ws.length > idx && ws[idx] == w)
1585                 ws[idx] = null;
1586             unlockRunState(rs, rs & ~RSLOCK);
1587         }
1588         long c;                                       // decrement counts
1589         do {} while (!U.compareAndSwapLong
1590                      (this, CTL, c = ctl, ((AC_MASK & (c - AC_UNIT)) |
1591                                            (TC_MASK & (c - TC_UNIT)) |
1592                                            (SP_MASK & c))));
1593         if (w != null) {
1594             w.qlock = -1;                             // ensure set
1595             w.transferStealCount(this);
1596             w.cancelAll();                            // cancel remaining tasks
1597         }
1598         for (;;) {                                    // possibly replace
1599             WorkQueue[] ws; int m, sp;
1600             if (tryTerminate(false, false) || w == null || w.array == null ||
1601                 (runState & STOP) != 0 || (ws = workQueues) == null ||
1602                 (m = ws.length - 1) < 0)              // already terminating
1603                 break;
1604             if ((sp = (int)(c = ctl)) != 0) {         // wake up replacement
1605                 if (tryRelease(c, ws[sp & m], AC_UNIT))
1606                     break;
1607             }
1608             else if (ex != null && (c & ADD_WORKER) != 0L) {
1609                 tryAddWorker(c);                      // create replacement
1610                 break;
1611             }
1612             else                                      // don't need replacement
1613                 break;
1614         }
1615         if (ex == null)                               // help clean on way out
1616             ForkJoinTask.helpExpungeStaleExceptions();
1617         else                                          // rethrow
1618             ForkJoinTask.rethrow(ex);
1619     }

1630         long c; int sp, i; WorkQueue v; Thread p;
1631         while ((c = ctl) < 0L) {                       // too few active
1632             if ((sp = (int)c) == 0) {                  // no idle workers
1633                 if ((c & ADD_WORKER) != 0L)            // too few workers

1642                 break;

1646             if (d == 0 && U.compareAndSwapLong(this, CTL, c, nc)) {
1647                 v.scanState = vs;                      // activate v
1648                 if ((p = v.parker) != null)
1649                     U.unpark(p);

1700      * Scans for and tries to steal a top-level task. Scans start at a
1701      * random location, randomly moving on apparent contention,
1702      * otherwise continuing linearly until reaching two consecutive
1703      * empty passes over all queues with the same checksum (summing
1704      * each base index of each queue, that moves on each steal), at
1705      * which point the worker tries to inactivate and then re-scans,
1706      * attempting to re-activate (itself or some other worker) if
1707      * finding a task; otherwise returning null to await work.  Scans
1708      * otherwise touch as little memory as possible, to reduce
1709      * disruption on other scanning threads.

1714      */

1716         WorkQueue[] ws; int m;
1717         if ((ws = workQueues) != null && (m = ws.length - 1) > 0 && w != null) {

1774      * returns false if the worker should terminate.  If inactivating
1775      * w has caused the pool to become quiescent, checks for pool
1776      * termination, and, so long as this is not the only worker, waits
1777      * for up to a given duration.  On timeout, if ctl has not
1778      * changed, terminates the worker, which will in turn wake up
1779      * another worker to possibly repeat this process.
1780      *
1781      * @param w the calling worker

1783      * @return false if the worker should terminate

1787             return false;

1789             if ((ss = w.scanState) >= 0)
1790                 break;

1792                 r ^= r << 6; r ^= r >>> 21; r ^= r << 7;
1793                 if (r >= 0 && --spins == 0) {         // randomize spins
1794                     WorkQueue v; WorkQueue[] ws; int s, j; AtomicLong sc;

1796                         (j = pred & SMASK) < ws.length &&

1798                         (v.parker == null || v.scanState >= 0))
1799                         spins = SPINS;                // continue spinning
1800                 }

1802             else if (w.qlock < 0)                     // recheck after spins
1803                 return false;

1806                 int ac = (int)((c = ctl) >> AC_SHIFT) + (config & SMASK);
1807                 if ((ac <= 0 && tryTerminate(false, false)) ||

1813                     if (t > 2 && U.compareAndSwapLong(this, CTL, c, prevctl))
1814                         return false;                 // else use timed wait
1815                     parkTime = IDLE_TIMEOUT * ((t >= 0) ? 1 : 1 - t);
1816                     deadline = System.nanoTime() + parkTime - TIMEOUT_SLOP;

1824                     U.park(false, parkTime);

2149     /**
2150      * Returns a cheap heuristic guide for task partitioning when
2151      * programmers, frameworks, tools, or languages have little or no
2152      * idea about task granularity.  In essence, by offering this
2153      * method, we ask users only about tradeoffs in overhead vs
2154      * expected throughput and its variance, rather than how finely to
2155      * partition tasks.
2156      *
2157      * In a steady state strict (tree-structured) computation, each
2158      * thread makes available for stealing enough tasks for other
2159      * threads to remain active. Inductively, if all threads play by
2160      * the same rules, each thread should make available only a
2161      * constant number of tasks.
2162      *
2163      * The minimum useful constant is just 1. But using a value of 1
2164      * would require immediate replenishment upon each steal to
2165      * maintain enough tasks, which is infeasible.  Further,
2166      * partitionings/granularities of offered tasks should minimize
2167      * steal rates, which in general means that threads nearer the top
2168      * of computation tree should generate more than those nearer the
2169      * bottom. In perfect steady state, each thread is at
2170      * approximately the same level of computation tree. However,
2171      * producing extra tasks amortizes the uncertainty of progress and
2172      * diffusion assumptions.
2173      *
2174      * So, users will want to use values larger (but not much larger)
2175      * than 1 to both smooth over transient shortages and hedge
2176      * against uneven progress; as traded off against the cost of
2177      * extra task overhead. We leave the user to pick a threshold
2178      * value to compare with the results of this call to guide
2179      * decisions, but recommend values such as 3.
2180      *
2181      * When all threads are active, it is on average OK to estimate
2182      * surplus strictly locally. In steady-state, if one thread is
2183      * maintaining say 2 surplus tasks, then so are others. So we can
2184      * just use estimated queue length.  However, this strategy alone
2185      * leads to serious mis-estimates in some non-steady-state
2186      * conditions (ramp-up, ramp-down, other stalls). We can detect
2187      * many of these by further considering the number of "idle"
2188      * threads, that are known to have zero queued tasks, so
2189      * compensate by a factor of (#idle/#active) threads.
2190      */
2191     static int getSurplusQueuedTaskCount() {
2192         Thread t; ForkJoinWorkerThread wt; ForkJoinPool pool; WorkQueue q;
2193         if (((t = Thread.currentThread()) instanceof ForkJoinWorkerThread)) {
2194             int p = (pool = (wt = (ForkJoinWorkerThread)t).pool).
2195                 config & SMASK;
2196             int n = (q = wt.workQueue).top - q.base;
2197             int a = (int)(pool.ctl >> AC_SHIFT) + p;
2198             return n - (a > (p >>>= 1) ? 0 :
2199                         a > (p >>>= 1) ? 1 :
2200                         a > (p >>>= 1) ? 2 :
2201                         a > (p >>>= 1) ? 4 :
2202                         8);
2203         }
2204         return 0;
2205     }

2226         }

2228         if ((rs & STOP) == 0) {
2229             if (!now) {                           // check quiescence
2230                 for (long oldSum = 0L;;) {        // repeat until stable
2231                     WorkQueue[] ws; WorkQueue w; int m, b; long c;
2232                     long checkSum = ctl;
2233                     if ((int)(checkSum >> AC_SHIFT) + (config & SMASK) > 0)
2234                         return false;             // still active workers
2235                     if ((ws = workQueues) == null || (m = ws.length - 1) <= 0)
2236                         break;                    // check queues
2237                     for (int i = 0; i <= m; ++i) {
2238                         if ((w = ws[i]) != null) {
2239                             if ((b = w.base) != w.top || w.scanState >= 0 ||
2240                                 w.currentSteal != null) {
2241                                 tryRelease(c = ctl, ws[m & (int)c], AC_UNIT);
2242                                 return false;     // arrange for recheck
2243                             }
2244                             checkSum += b;
2245                             if ((i & 1) == 0)
2246                                 w.qlock = -1;     // try to disable external
2247                         }
2248                     }
2249                     if (oldSum == (oldSum = checkSum))
2250                         break;
2251                 }
2252             }
2253             if ((runState & STOP) == 0) {
2254                 rs = lockRunState();              // enter STOP phase
2255                 unlockRunState(rs, (rs & ~RSLOCK) | STOP);
2256             }
2257         }

2259         int pass = 0;                             // 3 passes to help terminate
2260         for (long oldSum = 0L;;) {                // or until done or stable
2261             WorkQueue[] ws; WorkQueue w; ForkJoinWorkerThread wt; int m;
2262             long checkSum = ctl;
2263             if ((short)(checkSum >>> TC_SHIFT) + (config & SMASK) <= 0 ||
2264                 (ws = workQueues) == null || (m = ws.length - 1) <= 0) {
2265                 if ((runState & TERMINATED) == 0) {
2266                     rs = lockRunState();          // done
2267                     unlockRunState(rs, (rs & ~RSLOCK) | TERMINATED);
2268                     synchronized (this) { notifyAll(); } // for awaitTermination
2269                 }
2270                 break;
2271             }
2272             for (int i = 0; i <= m; ++i) {
2273                 if ((w = ws[i]) != null) {
2274                     checkSum += w.base;
2275                     w.qlock = -1;                 // try to disable
2276                     if (pass > 0) {
2277                         w.cancelAll();            // clear queue
2278                         if (pass > 1 && (wt = w.owner) != null) {
2279                             if (!wt.isInterrupted()) {
2280                                 try {             // unblock join
2281                                     wt.interrupt();
2282                                 } catch (Throwable ignore) {
2283                                 }
2284                             }
2285                             if (w.scanState < 0)
2286                                 U.unpark(wt);     // wake up
2287                         }
2288                     }
2289                 }
2290             }
2291             if (checkSum != oldSum) {             // unstable
2292                 oldSum = checkSum;
2293                 pass = 0;
2294             }
2295             else if (pass > 3 && pass > m)        // can't further help
2296                 break;
2297             else if (++pass > 1) {                // try to dequeue
2298                 long c; int j = 0, sp;            // bound attempts
2299                 while (j++ <= m && (sp = (int)(c = ctl)) != 0)
2300                     tryRelease(c, ws[sp & m], AC_UNIT);
2301             }
2302         }
2303         return true;
2304     }

2309      * Full version of externalPush, handling uncommon cases, as well
2310      * as performing secondary initialization upon the first
2311      * submission of the first task to the pool.  It also detects
2312      * first submission by an external thread and creates a new shared
2313      * queue if the one at index if empty or contended.
2314      *
2315      * @param task the task. Caller must ensure non-null.
2316      */
2317     private void externalSubmit(ForkJoinTask<?> task) {
2318         int r;                                    // initialize caller's probe
2319         if ((r = ThreadLocalRandom.getProbe()) == 0) {
2320             ThreadLocalRandom.localInit();
2321             r = ThreadLocalRandom.getProbe();
2322         }
2323         for (;;) {
2324             WorkQueue[] ws; WorkQueue q; int rs, m, k;
2325             boolean move = false;
2326             if ((rs = runState) < 0) {
2327                 tryTerminate(false, false);     // help terminate
2328                 throw new RejectedExecutionException();
2329             }
2330             else if ((rs & STARTED) == 0 ||     // initialize
2331                      ((ws = workQueues) == null || (m = ws.length - 1) < 0)) {
2332                 int ns = 0;
2333                 rs = lockRunState();
2334                 try {
2335                     if ((rs & STARTED) == 0) {
2336                         U.compareAndSwapObject(this, STEALCOUNTER, null,
2337                                                new AtomicLong());
2338                         // create workQueues array with size a power of two
2339                         int p = config & SMASK; // ensure at least 2 slots
2340                         int n = (p > 1) ? p - 1 : 1;
2341                         n |= n >>> 1; n |= n >>> 2;  n |= n >>> 4;
2342                         n |= n >>> 8; n |= n >>> 16; n = (n + 1) << 1;
2343                         workQueues = new WorkQueue[n];
2344                         ns = STARTED;
2345                     }
2346                 } finally {
2347                     unlockRunState(rs, (rs & ~RSLOCK) | ns);
2348                 }
2349             }
2350             else if ((q = ws[k = r & m & SQMASK]) != null) {
2351                 if (q.qlock == 0 && U.compareAndSwapInt(q, QLOCK, 0, 1)) {
2352                     ForkJoinTask<?>[] a = q.array;
2353                     int s = q.top;
2354                     boolean submitted = false; // initial submission or resizing
2355                     try {                      // locked version of push
2356                         if ((a != null && a.length > s + 1 - q.base) ||
2357                             (a = q.growArray()) != null) {
2358                             int j = (((a.length - 1) & s) << ASHIFT) + ABASE;
2359                             U.putOrderedObject(a, j, task);
2360                             U.putOrderedInt(q, QTOP, s + 1);
2361                             submitted = true;
2362                         }
2363                     } finally {
2364                         U.compareAndSwapInt(q, QLOCK, 1, 0);
2365                     }
2366                     if (submitted) {
2367                         signalWork(ws, q);
2368                         return;
2369                     }
2370                 }
2371                 move = true;                   // move on failure
2372             }
2373             else if (((rs = runState) & RSLOCK) == 0) { // create new queue
2374                 q = new WorkQueue(this, null);
2375                 q.hint = r;
2376                 q.config = k | SHARED_QUEUE;
2377                 q.scanState = INACTIVE;
2378                 rs = lockRunState();           // publish index
2379                 if (rs > 0 &&  (ws = workQueues) != null &&
2380                     k < ws.length && ws[k] == null)
2381                     ws[k] = q;                 // else terminated
2382                 unlockRunState(rs, rs & ~RSLOCK);
2383             }
2384             else
2385                 move = true;                   // move if busy
2386             if (move)
2387                 r = ThreadLocalRandom.advanceProbe(r);
2388         }
2389     }
