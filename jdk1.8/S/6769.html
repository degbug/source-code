  20 /**
  21  * JPEGEncodeParam encapsulates tables and options necessary to
  22  * control encoding of JPEG data streams.  Parameters are either set
  23  * explicitly by the application for encoding, or read from another
  24  * JPEG header.
  25 
  26  * When working with BufferedImages, the codec will attempt to match
  27  * the encoded JPEG COLOR_ID with the ColorModel in the BufferedImage.
  28  * This is not always possible (the default mappings are listed
  29  * below).  In cases where unsupported conversions are required (or
  30  * odd image colorspaces are in use) the user must either convert the
  31  * image data to a known ColorSpace or encode the data from a raster.
  32  * When encoding rasters no colorspace adjustments are made, so the
  33  * user must do any conversions required to get to the encoded
  34  * COLOR_ID.
  35 
  36  * The COLOR_ID for the encoded images is used to control the JPEG
  37  * codec's inital values for Huffman and Quantization Tables as well
  38  * as subsampling factors. It is also used to determine what color
  39  * conversion should be performed to obtain the best encoding.
  40 
  41  * Note: The color ids described herein are simply enumerated values
  42  * that influence data processing by the JPEG codec.  JPEG compression
  43  * is, by definition, color blind.  These values are used as hints when
  44  * compressing JPEG data.  Through these values the JPEG codec can
  45  * perform some default rotation of data into spaces that will aid in
  46  * getting better compression ratios.

  47 
  48  * Example behaviour is described below.  Since these mappings are
  49  * likely to change in the future it is strongly recommended that you
  50  * make use of the @see JPEGImageEncoder.getDefaultParamBlock calls
  51  * and check the encodedColorID for your particular BufferedImage.
  52 
  53  * In extreme cases is may be necessary for the user to convert the
  54  * image to the desired colorspace, and encode it from a Raster.  In
  55  * this case the API programmer must specify the colorID of the data
  56  * in the Raster and no color conversion will take place.
  57  

  58  ENCODING:
  59 
  60  BufferedImage Type/Instance        JPEG (Encoded) Color ID
  61  ========================       =======================
  62  TYPE_GRAY                      COLOR_ID_GRAYSCALE
  63  TYPE_RGB                       COLOR_ID_YCbCr
  64  TYPE_YCbCr                     COLOR_ID_YCbCr
  65  TYPE_YCbCr/CS_PYCC             COLOR_ID_PYCC
  66  TYPE_CMYK                      COLOR_ID_CMYK
  67  TYPE_RGB   (w/ alpha)          COLOR_ID_YCbCrA
  68  TYPE_YCbCr (w/ alpha)          COLOR_ID_YCbCrA
  69  TYPE_YCbCr/CS_PYCC (w/ alpha)  COLOR_ID_PYCCA
  70  ** Any Other **                COLOR_ID_UNKNOWN
  71  
  72 
  73  * When the user wants more control than the BufferedImage conversions
  74  * provide, the user must encode the data from a Raster. In this case
  75  * the data undergoes no color conversion at all. It is the user's
  76  * responsiblity to perform the desired conversions.

  77 
  78  * If you intend to write a JFIF image (by including the APP0_MARKER)
  79  * the encoded COLOR_ID must be one of: COLOR_ID_UNKNOWN,
  80  * COLOR_ID_GRAYSCALE, COLOR_ID_YCbCr, or COLOR_ID_CMYK. In all other
  81  * instances an ImageformatException will be thrown.

  82 
  83  * IMPORTANT: an Alpha RGB BufferedImage will not map to a
  84  * valid JFIF stream, you must strip off the alpha prior to encoding
  85  * if you want a JFIF file.  If the APP0 marker is set and you do not
  86  * strip off the Alpha, an ImageFormatException will be thrown.
  87  * 
  88  * Note that the classes in the com.sun.image.codec.jpeg package are not
  89  * part of the core Java APIs.  They are a part of Sun's JDK and JRE
  90  * distributions.  Although other licensees may choose to distribute these
  91  * classes, developers cannot depend on their availability in non-Sun
  92  * implementations.  We expect that equivalent functionality will eventually
  93  * be available in a core API or standard extension.
  94  * 
  95  */
  96

 232         /**
 233          * This creates new Quantization tables that replace the currently
 234          * installed Quantization tables.  It also updates the Component
 235          * QTable mapping to the default for the current encoded COLOR_ID.
 236 
 237          * The Created Quantization table varies from very high
 238          * compression, very low quality, (0.0) to low compression, very
 239          * high quality (1.0) based on the quality parameter.

 240 
 241          * At a quality level of 1.0 the table will be all 1's which will
 242          * lead to no loss of data due to quantization (however chrominace
 243          * subsampling, if used, and roundoff error in the DCT will still
 244          * degrade the image some what).

 245 
 246          * This is a linear manipulation of the standard Chrominance
 247          * Q-Table.

 248 
 249          * 
Some guidelines: 0.75 high quality

 250          *                 0.5  medium quality
 251          *                 0.25 low quality
 252          * 
 253          * @param quality 0.0-1.0 setting of desired quality level.
 254          * @param forceBaseline force baseline quantization table
 255          */
 256         public void setQuality(float quality, boolean forceBaseline );
